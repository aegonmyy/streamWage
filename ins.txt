The error is clear — `eth_newFilter` is a stateful method and `rpc.hoodi.ethpandaops.io` is a load-balanced RPC that routes requests across multiple nodes. Stateful methods like `eth_newFilter` require hitting the same node every time because the filter state lives on a specific node — when your next request hits a different node it has no idea what filter you're talking about.

---

**Methods that have this problem:**
```
eth_newFilter
eth_newBlockFilter
eth_getFilterChanges
eth_getFilterLogs
eth_uninstallFilter
```

---

**The fix — stop using filters, use `eth_getLogs` instead:**

`eth_getLogs` is stateless — you pass the full query every time, no filter ID needed, works fine with load-balanced RPCs.

In viem/wagmi replace any `watchContractEvent` or filter-based calls with:

```js
const logs = await publicClient.getLogs({
  address: CONTRACT_ADDRESS,
  event: parseAbiItem('event TermsProposed(address indexed worker, ...)'),
  fromBlock: DEPLOY_BLOCK,
  toBlock: 'latest'
})
```

Or if you're using wagmi's `useWatchContractEvent` — replace it with a polling approach:

```js
// Poll every 30 seconds instead of using a filter
const { data: logs } = useReadContract({
  ...
  query: { refetchInterval: 30_000 }
})
```

---

**Alternative — use a different RPC:**

If you need real-time event watching, use an RPC that supports stateful methods:

- Alchemy — free tier, supports Hoodi testnet
- Infura — same
- QuickNode — same

These give you a dedicated endpoint that always hits the same node so stateful methods work fine. For Hoodi specifically Alchemy has it under their testnet options.
